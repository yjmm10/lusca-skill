# lusca-paper-read — 论文精读与批判性评估

**核心理念：读懂一篇论文，不是复述它说了什么，而是判断它为什么重要、哪里可信、哪里可疑、对我有什么用。**

输入一篇论文（PDF / `.tex` / `.md` / arXiv ID / URL / 粘贴文本），按**六维精读框架**逐层
拆解，再用**五个批判性视角**（含反方最强论证）压力测试，最终落盘一份结构化阅读笔记。
立足**读者视角**：目标是让自己（读者）真正吃透这篇论文并留下可复用的笔记，而非给作者写
referee report 或 accept/reject 决定。

## 触发条件

- 明确提及：读论文、精读、帮我看看这篇论文、这篇讲了啥、解读一下这篇、值不值得读、阅读笔记

- 斜杠指令：`/lusca-paper-read`、`/读论文`
- 提供了 arXiv ID / 论文 URL / PDF 路径并希望理解其内容

**不触发**：
- 只想检索相关论文（→ `lusca-paper-search`）
- 已有初稿要润色（→ `lusca-paper-polish`）
- 只想知道一个已知事实 / DOI 的详情（直接 WebFetch 即可，无需精读流程）
- 要给作者写 referee report 用于投稿决策（本技能是读者视角；可参考产出自行改写）

## 输入

论文来源**从 `$ARGUMENTS` 与上下文自动解析**；仅当完全找不到论文时，**追问一次**论文位置
（读论文没有输入则无法进行，这是唯一允许的追问）：

- **文件路径**：`/path/to/paper.pdf`、`./papers/foo.tex`、`bar.md` —— 用 `Read`（PDF 支持分页）
- **arXiv ID**：`2310.12345` 或 `arXiv:2310.12345` —— 优先 `WebFetch` 抓 `arxiv.org/html/<id>`
  （全文 HTML，最适合精读，且便于提取图表 URL 嵌入笔记）；无 HTML 版则下载 PDF 用 `Read`
  分块读；`abs` 页面仅补元信息
- **URL**：任意论文页面 —— `WebFetch` 抓取
- **粘贴文本**：用户直接把标题 + 摘要 / 正文贴进对话 —— 直接读

**读多深**（自动推断，不追问）：
- 短文（abstract + 4 页以内正文）：全文通读
- 长文（>10 页 PDF）：分块读（`Read` 的 `pages` 参数，每次 5 页），必读 abstract / intro /
  method / results / conclusion，related work 与 appendix 按需扫读
- 用户指明"只看方法"/"只看实验"时，聚焦对应章节但仍输出完整笔记骨架

## 主流程

```
locate → read-full → six-dim-read → critique-lenses → note → save
```

1. **定位论文**：解析 `$ARGUMENTS`；找不到 → **追问一次**论文位置后继续
2. **读全文**：按"读多深"策略读完；长 PDF 分块（`Read` 的 `pages` 参数，每次 5 页）；
   留意图表并**建立图清单**——每张值得留下的图标明「Figure 号 → 内容 → 归位节（2.1 架构 / 2.2 模块 / 3.1 结果）」，据此下载到 `{slug}/imgs/`（命名 `figX_<内容>.jpg`）。**下载即承诺嵌入**：进了 `imgs/` 的图正文必须 `![]()` 引用，否则 Step 5 自检会报孤儿图
3. **六维精读（指导）**：`Read` `references/reading-dimensions.md`，按六维入口**审视**论文——
   这是解读时的内化指导，**不是笔记的输出结构**；不必逐维对号入座填表
4. **批判性视角（指导）**：`Read` `references/critique-lenses.md`，用五视角**找问题**——同样
   是指导而非输出结构；把真正有价值的发现**融入自然的批判性叙述**，不贴视角标签
5. **落盘笔记**：`Read` `assets/reading-note-template.md`，**严格对照模板逐节填写**（含 YAML
   frontmatter）；某节无内容写 `n/a` 并说明原因，**不删节**；写到
   `./outputs/lusca-paper-read/{slug}/{YYYYMMDDHHmmss}_{slug}.md`，原文图转存到同目录 `{slug}/imgs/`（用户指定路径时从其指定）。
   **落盘后跑图片完整性自检**：`python3 skills/lusca-paper-read/scripts/check_imgs.py outputs/lusca-paper-read/{slug}/`，
   校验 `imgs/` 每张图都被正文 `![]()` 引用；报孤儿图 → 补嵌到对应节（2.2 模块 / 3.1 结果），或确认无独立信息量后从 `imgs/` 删除——**不留下"下了不用"的孤儿图**
6. **交付**：笔记路径 + 一两句简要说明（论文一句话定位 + verdict）；**不内联重复展示完整笔记**——完整内容已落盘，再贴一遍是冗余噪声；附一句后续衔接建议（见 §后续衔接）

## 六维精读框架（摘要）

**定位**：六维是解读时的**审视入口**（指导你全面看论文），不是笔记的输出结构——笔记的批判性评估写自然叙述，不逐维对号入座。详见 `references/reading-dimensions.md`。

| 维度 | 回答的问题 | 读者收益 |
|------|-----------|----------|
| 1. 定位与贡献 | 研究问题是什么？核心 claim？与已有工作差在哪？ | 知道这篇在领域里的位置 |
| 2. 方法与设计 | 怎么做的？关键假设？数据/实验设置？ | 能判断方法是否靠谱、能否复用 |
| 3. 结果与证据 | 主要发现？证据强度？结论是否被数据支撑？ | 区分"证明了"与"暗示了" |
| 4. 逻辑与论证 | 论证链完整吗？有跳跃/过度泛化吗？反方最强反驳？ | 不被流畅写作带偏 |
| 5. 写作与呈现 | 清晰吗？图表自洽？notation 一致？ | 评估可信度的一个侧面信号 |
| 6. 可复用性与影响 | 对我有什么用？可借鉴的方法/数据/idea？值得引用吗？ | 把论文变成自己的资产 |

---

## 批判性视角（摘要）

**定位**：五视角是解读时的**挑刺方法**（指导你找问题），不是笔记的输出结构——发现融入批判性叙述，不贴视角标签、不对号入座。详见 `references/critique-lenses.md`。五个视角作为审视思路：

| 视角 | 做什么 |
|------|--------|
| 假设压力测试 | 找出论文成立所依赖的关键假设，逐个问"若不成立会怎样" |
| 反方最强论证 | 站在对立面，构造对论文核心 claim 最有力的反驳（吸收自 academic-paper-reviewer 的 Devil's Advocate） |
| cherry-picking / 确认偏误检测 | 检查是否只挑有利结果、隐藏不利实验、措辞引导 |
| "so what" 测试 | 若论文结论完全成立，对领域/实践到底改变了什么 |
| 替代解释 | 同样的结果有没有别的解释（混杂、选择偏倚、baseline 偏弱） |

## 输出与存放

- **目录结构**（每篇论文一个文件夹，图片放其下 `imgs/` 以保证在笔记中正常显示）：
  ```
  outputs/lusca-paper-read/{slug}/
    ├── {YYYYMMDDHHmmss}_{slug}.md   # 阅读笔记
    └── imgs/                         # 转存的原文图片（Figure/Table 图）
  ```
  - slug 取自论文标题或 arXiv ID（kebab-case），同时作为文件夹名；用户指定路径时从其指定
  - 示例：`outputs/lusca-paper-read/diffusion-policy-robotics/20260714143022_diffusion-policy-robotics.md` + `imgs/fig1_arch.jpg`
- **图片转存**：笔记引用的原文图一律**复制到 `{slug}/imgs/`**，正文用相对路径 `![](imgs/figX_xxx.jpg)` **直接展示**；严禁只写"原图见 …"文字路径（跨目录会断链且不可视）
- **模板**：`assets/reading-note-template.md`（复制即用）；笔记带 YAML frontmatter
  （`title`/`authors`/`year`/`venue`/`arxiv`/`doi`/`read_date`/`verdict`/`slug`/`tags`），
  便于后续按 verdict / tag 检索
- **交付精简**：落盘后只返回路径 + 一两句要点（论文定位 + verdict），**不内联重复展示完整笔记**——完整内容已写入文件，再贴一遍是冗余噪声
- **文末出处块**：正文结尾附一行出处（遵 CLAUDE.md「文档输出规范」）：`> 作者：lusca ｜ 版本：lusca-paper-read v<version> ｜ 出处：https://github.com/yjmm10/lusca-skill/tree/main/skills/lusca-paper-read`

## 阅读笔记结构

笔记结构与字段**以 `assets/reading-note-template.md` 为准**——生成时严格对照模板逐节填写。
元信息**只写在 YAML frontmatter**，正文不重复。主体节：

- **TL;DR**：2–3 句可独立传播的总结（核心 claim + 关键证据 + 主要 caveat）
- **1. 研究内容**：1.1 研究问题、痛点与动机（**研究问题 / 痛点 / 动机·重要性 / 领域定位** 四点） / 1.2 核心贡献 / 1.3 相关工作脉络（mermaid 流程图）
- **2. 方法概要**：方法路线、关键假设、数据/实验设置；**先一段话通讲整体处理流程**（模块组成 + 数据流转）；再逐模块**讲透实现原理**（2.2）；原文有伪代码/算法则**转述+逐步解读**（2.3）
  - 2.1 架构图（原文有架构图则嵌原图、**不再补 mermaid 重绘**；仅原文无图时才 mermaid 重绘简化）
  - 2.2 模块详解（逐模块**讲透实现原理**：输入→处理→输出 + 为什么这么设计 + 关键参数；**不是列参数清单**）
  - 2.3 算法 / 伪代码（**原文有则转述并对关键步骤逐步解读**；不要凭空生成原文没有的伪代码；原文无则 n/a）
- **3. 关键结果**：主要发现、证据强度、最支撑结论的证据；**每个重要实验都配原文图/表并对重点结果逐项解读**
  - 3.1 关键结果图 / 表（原文图转存到 `imgs/` 用 `![]()` 展示；表格一律 markdown 重现，原文表格是图片也转写）
- **4. 批判性评估与价值**（合并细分）：4.1 批判性评估（自然叙述，六维+五视角作内化指导）+ 4.2 Limitations 与复现性 + 4.3 可复用与后续
- **Verdict**：推荐深读 / 选读 / 跳过 + 理由

第 1–3 节是论文内容解读，事实层、**饱满不可简化**；第 4 节是读者评估。

## 模板使用约定

- **元信息只在 frontmatter**：title/authors/year/venue/arxiv/doi/source/read_date/verdict/slug/tags 写进 YAML；正文不再单列元信息节
- **逐节填写，不删节**：模板每一节（含小点）都要在笔记里出现；无内容写 `n/a` 并说明原因
- **frontmatter 必填**：`title`/`read_date`/`verdict`/`slug` 必填；其它字段无则留空字符串/空数组
- **批判性评估是自然叙述**：六维与五视角是**内化指导**，不作为固定输出结构对号入座、不贴框架标签；发现融入叙述；若反方论证有力可单独引出（非强制）；综合可信度必出
- **事实 vs 评估**：第 1–3 节（研究内容 / 方法概要 / 关键结果）是客观陈述；第 4 节（批判性评估与价值）是读者评估，用"评估："前缀或明确语气区分
- **论文解读饱满、评估节细分**：研究内容 / 方法概要 / 关键结果要写足；第 4 节合并批判性评估 + Limitations/复现性 + 可复用/后续三个小点，其中 4.2/4.3 每条一行不展开
- **相关工作用流程图**：1.3 节用 mermaid 流程图展示相关工作发展到本文的脉络，节点标"工作 + 作者年份"，箭头表示发展/启发关系，本文方法作为终点突出
- **mermaid 图：节点 ID 语义化、解读用内容不引用字母 ID**：mermaid 节点 ID 用**语义化 slug**（工作名/模块名的 kebab 或 snake-case，如 `doclayout_yolo`、`mask_head`），**不用 A/B/C 这类无意义字母**——字母只是语法占位符；图下的传承/分歧等解读文字一律用**节点内容（工作名+年份 / 模块名）**指代，**严禁出现 "A→B→C"、"DLAFormer（I）" 这类对节点字母 ID 的引用**，不应让读者回头对照图查字母。对 1.3 相关工作流程图与 2.1 架构图均适用
- **列表符号规范：并列用无序、分步用有序逐行、不堆 ①②③ / (1)(2)(3)**：并列要点一律用无序列表（`-`）逐行展开，**不要把 ①②③ 或 (1)(2)(3) 塞在同一段里**——要分点就分行写成列表项；**真正的步骤 / 阶段 / 递进（"分 N 步处理"、训练 / 推理流程、编码器多阶段聚合、特征金字塔路径等）一律用有序列表 `1.` `2.` 逐行写，不要用 (1)(2)(3) 内联塞在一段**；无递进的并列（多条贡献、多个假设、多个发现、多个警惕点）用无序 `-` 不用有序
- **段落可拆就拆，别压长段**：一段里若塞了多个语义上可独立并列的要点（多个机制 / 多个理由 / 多个条件 / 多个子能力），**拆成无序列表或分段**以提高扫读性；仅当语义连贯、确实构成一段完整论述时才保留整段。目标是源码可读、渲染清晰，而非把相关信息都压进一段
- **方法要讲透**：整体流程一段话通讲 + **逐模块讲透实现原理**（输入→处理→输出 + 设计理由 + 关键参数，不是列要点清单）+ 原文伪代码转述并逐步解读 + 抽象处配具体示例（数值/场景/对比）；要让没读过原文的人跟上
- **图表转存并嵌入（方法+结果都要配图）**：原文图一律**复制到 `{slug}/imgs/`**，正文用 `![](imgs/figX_xxx.jpg)` **直接展示**；arXiv HTML 等在线来源可用绝对 URL。**严禁只写"原图见 …"文字路径**——必须可视嵌入，否则断链不可视
- **有原图就不补 mermaid（避免重叠冲突）**：某节若已嵌入原文图（架构图 / 结果图等），**不再额外画 mermaid 重绘图**——两份重叠内容会冲突、徒增噪声；mermaid 仅用于原文无对应图、需自行归纳可视化的场景（典型如 1.3 相关工作脉络）。原文图与 mermaid 二选一，不并存
- **伪代码只转述原文已有的**：原文给出 Algorithm/伪代码时，完整转述并对关键步骤做解读（这步算什么、为什么）；**不要凭空生成原文没有的伪代码**；原文无显式算法写 n/a
- **表格尽量展示**：关键结果表格一律用 markdown 重现（原文表格是图片也转写并标"见原文 Table X"）；不止给结论，把支撑结论的关键数字摆出来
- **不确定必标**：没读懂的、臆测的，标 `(uncertain)`
- **数学公式用规范 LaTeX（行内 `$`、行间 `$$`，主动修复损坏公式）**：行内公式用单美元符 `$...$`（如 `$x_t$`、`$\alpha+\beta$`），独立成行的行间公式用 `$$...$$`（前后各空一行）。**主动修复原文转写中损坏的 LaTeX**——下标/上标用 `_`/`^`、希腊字母用 `\alpha \beta \gamma` 等标准宏、运算符用 `\times \cdot \sum \prod \frac{}{}` 而非裸字符、补全缺失的 `{}` 与定界符；OCR / HTML 转写产生的乱码（如 `x2`→`$x_2$`、`θ`→`$\theta$`、`a^b c`→`$a^{bc}$`）尽量还原为可正确渲染的 LaTeX，不要把损坏或半成品的公式原样塞进笔记。变量符号在叙述中首次出现也用行内 `$...$`，与论文 notation 保持一致
- **mermaid / 代码块等不渲染 LaTeX 的区域里不写公式表达**：mermaid 图（节点标签、边标注）与 ``` 代码块 / 伪代码块**不渲染 `$...$` / `$$...$$`**——LaTeX 定界符会被当字面字符显示、甚至会触发 mermaid 语法错误。这些区域里要表达数学 / 逻辑时**改用文字与简短符号**（mermaid 节点写"沿弦法向偏移"而非 `$d_i$`；伪代码写 `sigmoid(...)` 或"sigmoid"而非 `$\sigma$`）；正式公式放到正文用 `$$` / `$`（见上一条）。若 mermaid 节点或伪代码步骤必须含某公式，则**把公式拆到块外的正文 `$$`**，块内只留文字 / 纯符号步骤

## 质量底线

- **基于原文，不臆造**：每条评估尽量指到具体章节/图表/公式；读不清的地方明说"此处未读清"，绝不编造细节
- **核心内容精讲，复杂处举例**：方法 / 算法 / 核心机制要讲透——**整体流程先用一段话通讲**（模块组成 + 数据怎么流转），再说清每个模块"是什么、为什么这么设计、怎么做、起什么作用"，**讲透实现原理（输入→处理→输出）而非堆砌参数清单**；**原文已有的伪代码要转述+逐步解读，不要凭空臆造原文没有的算法**；表示过于抽象时配一个具体例子（数值、场景或对比），让没读过原文的人也能跟上
- **区分事实与判断**：论文陈述的内容用客观语气，读者评估用"评估："前缀，不混为一谈
- **批判要构造性地给**：指出问题的同时给出"若要复用/借鉴此工作，需先验证 X"
- **读者视角**：不产出 accept/reject 决定，不写 referee report 语气；笔记是给自己未来回看用的
- **诚实标注不确定**：对论文某部分理解不充分时，在笔记里标 `(uncertain)`，不假装读懂
- **图片零闲置**：`imgs/` 里每张图都必须在正文 `![]()` 出现至少一次；落盘后跑 `scripts/check_imgs.py` 自检，报孤儿图必须补嵌或删除——不允许"下载了却没嵌"
- **不替代检索**：精读中发现需要查 related work / 找更多同类论文时，引导用户用
  `lusca-paper-search`，而非自行 WebFetch 检索——本技能只读给定的一篇
