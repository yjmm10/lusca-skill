# 实验 / 评估范式

## 本节专属审稿维度

**控制变量 + 数字带前提 + 局限真诚**：对比须同数据集/同环境（否则无效）；引用的性能数字须带
前提条件（数据集/场景/密度）；局限性反思须坦诚（专设 Limitations 列/节，掩盖缺陷反遭严惩）。

## 组织范式：设置 → 结果 → 分析（隔离效应/反思）

### 变体 A：研究型（自己的实验）
1. **实验设置与评估协议**：数据/环境/基线/指标/训练协议（服务于可复现）。
2. **总体性能**：主指标绝对值 + 相对增益（配表/图）。
3. **消融/析因隔离效应**：拆解各模块/组件的贡献（凸创新有效）。
4. **稳健性与局限**：不同条件下的表现 + 诚实局限。

**常见毛病**：①只给结论不给支撑数字；②析因/消融铺陈过细挤占主结果；③报喜不报忧（关键不利结果藏起来）。

### 变体 B：综述型（归纳别人的实验）
1. **按场景分类归纳**：按应用层级/任务类型分小节。
2. **实验目标与算法汇总表**：`| 场景 | 技术 | 参考文献 | 目标/性能 |`。
3. **定量结果提炼**：提取关键性能提升数据（如"碰撞率降低 81.04%、感知人类似度提高 50%"）。
4. **局限性反思**：指出实验设置的缺陷（如"缺乏对原始数据的直接访问""未在真实云基础设施实测"）。

## 审稿人避坑（隐性严谨性事故）

- [通用] **对比未控制变量**：汇总表若不列"数据集/仿真环境"和"优化目标"，对比的两个算法不在
  同一数据集或不同信道环境下 → 审稿人判对比无效。**表头强制含数据集/环境 + 优化目标**，显式
  标注场景假设。综述型表头建议：`| 参考文献 | 算法/模型 | 数据集/环境 | 优化目标 | 主要指标 | Limitations |`。
- [通用] **性能数字脱离前提**：引用"精度 99.8%""碰撞率降低 81.04%"却不说在什么数据集/场景/
  密度下 → 脱离前提的数字在顶刊眼中无意义。**每个引用数字须带前提**（"在 NSL-KDD 上
  Acc=99.8""在 UE 密度>100 的 V2X 场景下碰撞率降 81.04%"）。
- [通用] **局限性避重就轻**：把现有方法夸得完美、藏起缺陷 → 审稿人最反感。顶级文章**专设
  Limitations 列/节**，坦诚指出（如"缺乏真实云基础设施实测""不进行模型篡改无法达最优"）——
  这种坦诚反而增加可信度。
- [研究型] **报喜不报忧**：关键不利结果藏起来 → 如实陈述，建议作者先做可信度自查。
- [通用] **数字与正文/表格矛盾**：正文写的数与表里对不上 → 逐项核对一致。

## 实验自检
- [ ] 研究型：主结果是否给量化（绝对值+相对）？消融是否凸出创新有效？有无报喜不报忧？
- [ ] 综述型：场景分类是否清晰？定量提炼是否标注来源与前提？局限反思是否具体？
- [ ] **控制变量**：对比是否同数据集/同环境？表头是否含数据集/环境 + 优化目标？
- [ ] **数字带前提**：每个引用数字是否标注数据集/场景/密度等前提？
- [ ] **局限真诚**：有无专设 Limitations？是否坦诚而非夸完美？
- [ ] 数字是否与正文/表格一致（不矛盾）？
