# 关联判定：问题 ↔ Skill

lusca 只记录**与 skill 强相关**的问题。本文件给出判定方法与正反例。这是 lusca 的**核心护栏**——决定一个问题有没有资格进 reflex 文档。

## 判定三步

对每个候选问题依次问，**任一步不通过即剔除**：

### 1. 可定位（Locatable）

问题能否指到**一个具体 skill**？

- ✅ 能说出 skill 名（如 `lusca-paper-search`、`paper-polish-skill`、`project-uav-skill`、`patent-skills`、`lusca` 自身）
- ❌ 问题是"模型本身能力不行""网络断了""我不熟悉这个领域""Markdown 渲染错位" → **剔除**（无 skill 归属）

### 2. 在职责内（In Scope）

问题是否落在该 skill 的**功能 / 触发 / 流程 / 输出 / 规范**范围内？

判定锚点——读该 skill 的 `SKILL.md` 的 description 与触发条件，问：**"修这个 skill 能解决/缓解这个问题吗？"**

- ✅ 能 → 通过
- ❌ 不能（问题在 skill 职责之外）→ **剔除**

**关键约束**：问题必须与该 skill 的本职强相关。

> **检索 skill 正反例（`lusca-paper-search`）**：
> - ✅ 相关：检索没被触发 / 触发词太窄；检索词构造不当导致结果偏；数据源选择策略错（该走 arXiv 却走了万方）；某个 `scripts/search_papers_by_*.py` 结果解析/去重/分页有问题；检索范围内 skill 内容（如 references/programmatic_api.md）有错。
> - ❌ 无关："检索完帮我写周报""把检索结果翻译成日语""检索结果帮我投稿"——这些是**检索之外的下游任务**，修检索 skill 解决不了，**剔除**。

### 3. 可归因（Attributable）

问题能否映射到 skill 的**某个组件**？

| 组件 | 典型对应 |
|------|---------|
| `frontmatter` | name / description / 触发词 / 路由 / allowed-tools |
| `prompts/` | 某步流程的指令、顺序、缺失 |
| `references/` | 某条规范缺失、有误、未被执行 |
| `scripts/` | 脚本参数、输出格式、数据源对接、降级逻辑 |
| 护栏 | 边界、禁止项、不越界规则 |
| 协作 | 跨 skill 分工、交接点、输出对接 |

- ✅ 能指到组件 → 通过，进入六维分析
- ❌ 只能泛泛说"这个 skill 不好用"、指不到组件 → 降级为**观察项**，不进改进建议（或先补全到组件级再记）

## 剔除后的处理

剔除的问题**不写入** reflex 文档，但在对话中告知用户：

> 「以下问题超出 lusca 记录范围，未归档：[问题] —— 理由：[未通过第 N 步：说明]」

## 边界情况

| 情况 | 处理 |
|------|------|
| 问题横跨多个 skill | 都保留，归属维度标注主责 skill + 协作 skill |
| 问题是"用户误用"而非 skill 缺陷 | 标注「非 skill 缺陷」，记录现象与澄清建议，不进改进建议 |
| 问题证据不足（只出现一次、无复现） | 可记录，但影响标「待证实」，改进建议标「待复现确认」 |
| 用户坚持记录被剔除的问题 | 记录到 reflex 文档的「观察项」区（不进改进建议），注明超出常规判定 |
