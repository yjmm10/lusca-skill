# 六维分析框架

每个通过关联判定的问题，依序从六维拆解。目标是把"这里不对"变成"改哪个 skill 的哪个文件的哪一节、怎么改、怎么验"。

## 维度速查

| 维度 | 一句话 | 必产出 |
|------|--------|--------|
| 场景 Scenario | 何时复现 | 触发条件 + 频率 |
| 根因 Root Cause | 出在哪一层 | 层级归类 |
| 影响 Impact | 后果多重 | 类型 + 范围 |
| 归属 Ownership | 改谁 | skill + 文件 + 小节 |
| 改进 Fix | 怎么改 | 改法 + 类型（最小） |
| 验证 Verify | 怎么验 | 手测 / 回归 / 补 eval |

---

## 1. 场景 Scenario

- **触发条件**：什么用户输入、什么文件类型、什么模式下复现？给出可复现的最小情形。
- **频率**：偶发 / 常见 / 必现；依据（本会话出现几次、是否每次都触发）。
- 作用：判断是否值得优先改。

## 2. 根因 Root Cause

归到**一层**（只选主因）：

| 层级 | 典型症状 |
|------|---------|
| 路由 Routing | 该触发没触发 / 不该触发却触发；触发词缺失或过宽 |
| 流程 Process | 某步漏做、顺序错、步骤断裂；prompt 指令不清或冲突 |
| 规范 Reference | 缺某条规范、规范有误、规范未被执行 |
| 脚本 Script | `scripts/` 参数/输出/数据源/降级逻辑有问题 |
| 边界 Guardrail | 越界（改了不该改的）、该拒绝没拒绝、边界未定义 |
| 协作 Inter-skill | 与其他 skill 分工不清、交接丢失、输出对不上下游 |

## 3. 影响 Impact

- **类型**（选最重的一项）：
  - 阻塞：任务无法继续
  - 降质：能完成但产出质量下降
  - 误导：给出错误结论/方向
  - 体验：多余追问、啰嗦、不一致
- **范围**：影响单个用户任务 / 影响该 skill 所有调用 / 影响下游 skill

## 4. 归属 Ownership

- **主责 skill** + **文件** + **小节**，精确到可定位。
  - 示例：`lusca-paper-polish` → `prompts/diagnose.md` → §实验章小节诊断
  - 示例：`lusca-paper-search` → `scripts/search_papers_by_arxiv.py` → 分页解析
- 横跨多 skill 时另列**协作 skill**，标注交接点。

## 5. 改进 Fix

- **改法**：具体到"加一段 / 改一句 / 补一个 reference / 调一个触发词 / 修脚本某段"。
- **类型**：补充（Add）/ 修正（Fix）/ 重构（Refactor）/ 新增（New file or mode）。
- **最小改动原则**：优先最小、可逆的改动；大改需在建议里标注风险。
- **不执行**：只写建议，状态「待确认」。改动落在 `skills/<name>/` 源，经 link 脚本同步。

## 6. 验证 Verify

- **手测**：最小复现用例（输入 → 期望输出）。
- **回归**：触发词 / 路由回归（改了路由层时必做）。
- **补 eval**：建议新增/调整 eval 用例（本工作区 eval 框架待建，作为后续建议登记）。

---

## 质量门槛

- 六维**缺一不可**；空泛的维度（如根因 = "设计不好"、改进 = "优化一下"）不合格，重写。
- 改进 Fix 必须能直接交给维护者执行，无需再猜。
- 若某问题无法填满六维，降级为「观察项」，不进改进建议清单。
